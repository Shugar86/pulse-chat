## Как внести изменения

1. **Форкните** репозиторий (если работаете не в основном).
2. **Создайте ветку** от `main`:

   ```bash
   git checkout -b feat/краткое-описание
   ```

3. **Внесите изменения.** Один PR — одна ответственность.
4. **Запустите проверки** в затронутых пакетах:

   ```bash
   pnpm install
   pnpm db:generate   # если менялась схема Prisma
   pnpm typecheck
   pnpm lint
   pnpm test
   ```

5. **Закоммитьте** с понятным сообщением:

   ```bash
   git commit -m "feat(api): добавил health check endpoint"
   ```

6. **Откройте Pull Request** в `main` с кратким описанием:
   - Что изменилось и зачем.
   - Как проверить вручную.
   - Какие риски или ограничения есть.

## Настройка окружения для разработки

```bash
# 1. Клонировать репозиторий
git clone git@github.com:Shugar86/pulse-chat.git
cd pulse-chat

# 2. Установить зависимости
pnpm install

# 3. Подготовить env-файлы
cp .env.example .env
cp .env.example services/api/.env
cp .env.example apps/mobile/.env
# отредактируйте секреты и URL

# 4. Поднять postgres, redis, wireguard
docker compose up -d

# 5. Применить схему БД
pnpm db:generate
pnpm db:push

# 6. Запустить бэкенд
pnpm -C services/api dev

# 7. В другом окне запустить мобильное приложение
pnpm -C apps/mobile start
```

## Стиль коммитов

Мы используем Conventional Commits:

| Префикс | Когда использовать |
|---------|-------------------|
| `feat:` | Новая фича |
| `fix:` | Исправление бага |
| `docs:` | Документация |
| `chore:` | Рутина, обновление зависимостей |
| `refactor:` | Рефакторинг без изменения поведения |
| `test:` | Тесты |
| `security:` | Исправления, связанные с безопасностью |

Примеры:

```bash
git commit -m "feat(api): add tenant-scoped message search"
git commit -m "fix(mobile): prevent double call accept"
git commit -m "docs: update deployment instructions"
```

## Стиль кода

- **TypeScript everywhere.** Избегайте `any` без явной причины.
- **Минимальные изменения** — чиним баг, не переписываем соседний код.
- **Безопасность в приоритете:** не логируйте секреты, не передавайте приватные ключи на сервер.
- **Локализация:** новые пользовательские строки добавляйте через i18next.
- **Форматирование:** следуйте существующему стилю проекта. Линтер — авторитет.

## Definition of Done

Перед тем как отправить PR, убедитесь, что:

- [ ] Код компилируется (`pnpm typecheck`).
- [ ] Линтер не ругается (`pnpm lint`).
- [ ] Тесты проходят (`pnpm test`).
- [ ] `.env` и секреты не попали в коммит.
- [ ] PR описывает, что изменилось и как проверить.
- [ ] Документация обновлена, если изменения влияют на запуск или API.

---

## Безопасность

- Никогда не коммитьте `.env`, `*.pem`, ключи, токены и пароли.
- Приватные ключи VPN остаются на устройстве пользователя.
- Если нашли утечку секрета — немедленно сообщите владельцу репозитория и отзовите ключ.
